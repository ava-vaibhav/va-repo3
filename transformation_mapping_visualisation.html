Overview

This integration acts as a secure pass-through API endpoint. A caller — typically a front-end application or another service — sends a single HTTP request containing their own bearer token, the tax year they want issuers for, and a unique request identifier. The integration validates that all three inputs are present, then calls the Avalara 1099 service on the caller's behalf to retrieve a paginated list of registered tax issuers for that year. The full response is returned synchronously, so the caller receives the issuer data directly in the HTTP reply without any polling.

Field mappings

  • The caller's bearer token is forwarded directly as the authorization credential for the Avalara API call, so the request runs under the caller's own identity.

    bearerToken → direct copy → Authorization header

    The value must already include the Bearer prefix (e.g. Bearer eyJ…). The integration does not add the prefix itself.

  • The requested tax year is turned into a server-side filter so that Avalara returns only issuers registered for that specific year, rather than all issuers.

    taxYear → format("taxYear eq {taxYear}") → $filter query param

    The filter follows OData v4 syntax, which is the query language Avalara's 1099 API uses. Sending an invalid year value (e.g. a string) will result in an Avalara API error.

  • The caller's unique request identifier is passed through to Avalara unchanged, making it possible to trace a specific request end-to-end across both systems using the same ID.

    xCorrelationId → direct copy → X-Correlation-Id header

Notes

Validation — all three inputs are required

The integration checks for bearerToken, taxYear, and xCorrelationId before making any outbound call. If any field is missing or null, the webhook returns an error immediately without calling Avalara.

Bearer token format

The bearerToken field must include the Bearer prefix as part of its value. The integration forwards it verbatim to the Authorization header — it does not prepend the prefix automatically.

Error handling — Avalara API failures

If Avalara returns any non-successful HTTP status (4xx or 5xx), the integration logs the status code and response body, then re-throws the error. The caller receives a failed response with the Avalara error details. There is no automatic retry.

Synchronous response

The webhook is configured in sync mode, meaning the HTTP connection to the caller is held open until the Avalara API call completes. The caller receives the issuer list (or an error) directly in the response body — no polling or webhooks back to the caller are required.

Pagination

Avalara may return a nextLink field in the response if there are more issuers than fit in a single page. The integration returns this link to the caller as-is; following it for subsequent pages is the caller's responsibility.
